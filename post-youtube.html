How I Helped a Friend Build His Own YouTube Analytics Tool

A friend just started making YouTube videos and wanted to track his own analytics without relying on YouTube Studio. What followed was a crash course in APIs, Python, and why the setup always takes longer than the actual code.

A friend of mine recently started a YouTube channel. Like most new creators, he was obsessed with the numbers, views, watch time, click-through rates, but found YouTube Studio's interface clunky for the kind of analysis he wanted to do. He'd tried building his own dashboard but kept hitting a wall at the same place: the credentials.

He reached out and I walked him through the whole thing, start to finish, doing it in front of him so he could actually understand what was happening rather than just copy-pasting commands he didn't understand.

Where Everyone Gets Stuck

The YouTube Data API is well-documented, but the OAuth setup is the part that trips people up every time. You need to create a project in Google Cloud Console, enable the YouTube Data API, create credentials, download the JSON file, and place it in the right directory before any of your Python code will even talk to Google's servers.

That credentials file is where he was stuck. He had the code, he had Python installed, but the auth flow kept failing because the file wasn't in the right place and the scopes weren't set correctly. Five minutes of looking at it together and we had it working.

"Most people don't fail because the code is too hard. They fail because nobody showed them what the setup actually looks like when it's done right."

The Full Setup

Once the credentials were sorted, we installed Python properly and went through each pip install one at a time so he understood what each library was doing, google-auth for authentication, google-api-python-client for making the actual API calls, and a couple others for data handling and display.

Then we wrote a script that pulled his channel stats and video performance data and displayed it cleanly in the terminal. Nothing fancy, just the numbers he actually cared about, formatted in a way that was easy to read at a glance.

The last thing we set up was a .bat file, a simple batch script on Windows that lets him run the whole thing with a double-click instead of opening a terminal and typing commands every time. That single quality-of-life addition made it something he'd actually use daily rather than once and forget.

API Safety, The Part People Skip

One thing I made sure to go through carefully was API key security. It's easy to hardcode credentials directly into a script when you're just getting something working, and a lot of tutorials do exactly that. The problem is that if you ever share your code, push it to GitHub, or even just leave it on a shared computer, your keys are exposed.

We set up a .env file to store the sensitive credentials separately from the code, added it to .gitignore so it would never accidentally get committed, and talked through what to do if a key ever did get compromised, how to revoke it, generate a new one, and rotate it without breaking the app.

It adds maybe ten minutes to the setup. It's worth it every time.

Why I Enjoy Doing This

The technical part of that afternoon took maybe two hours. But what I find genuinely satisfying about helping people with this stuff isn't the code, it's watching someone go from "I don't understand any of this" to "I could build the next part myself." That shift in confidence is the whole point.

Most of the barriers to building things like this aren't intelligence or ability. They're just unfamiliarity. Once you've seen how it's supposed to look when it works, the whole thing demystifies fast.

If you're trying to build something and keep hitting the same wall, reach out. Sometimes you just need someone to sit down and do it with you.
